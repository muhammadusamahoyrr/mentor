# `migration.sql` vs `validation/noteSchema.js`

These two files look similar (both mention "fields" and "required") but they do
**completely different jobs** at **different times**. This note explains both.

---

## File 1: `prisma/migrations/.../migration.sql` — builds the DATABASE tables

This is **SQL** that creates the actual tables in the database:

```sql
-- CreateTable
CREATE TABLE "Doctor" (
    "id" INTEGER NOT NULL PRIMARY KEY AUTOINCREMENT,
    "name" TEXT NOT NULL,
    "email" TEXT NOT NULL,
    "specialization" TEXT,
    "createdAt" DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- CreateTable
CREATE TABLE "Note" (
    "id" INTEGER NOT NULL PRIMARY KEY AUTOINCREMENT,
    "title" TEXT NOT NULL,
    "content" TEXT NOT NULL,
    "createdAt" DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    "updatedAt" DATETIME NOT NULL,
    "doctorId" INTEGER NOT NULL,
    CONSTRAINT "Note_doctorId_fkey" FOREIGN KEY ("doctorId") REFERENCES "Doctor" ("id") ON DELETE CASCADE
);
```

**In plain English:**

- "Make a `Doctor` table with these columns."
- "Make a `Note` table with these columns."
- `FOREIGN KEY ("doctorId") REFERENCES "Doctor"` = **this is the relationship** —
  every note must point to a real doctor.

**Key facts:**

- **When does it run?** Once — when you ran `npx prisma migrate dev`. It's the
  "construction blueprint" for the database. It does **not** run on every request.
- **Who wrote it?** You didn't. **Prisma auto-generated it** from `schema.prisma`.
  That is the whole point of migrations: you describe models in `schema.prisma`,
  Prisma writes the SQL.
- **Level:** the **database**. It shapes where data is *stored*.

> Think of it as: **building the shelves in a warehouse.** You build them once.
> Every column is a labeled slot.

---

## File 2: `src/validation/noteSchema.js` — checks INCOMING requests

This is **JavaScript (Zod)** that inspects data arriving from the user:

```js
const createNoteSchema = z.object({
  doctorId: z.coerce.number().int().positive(),                     // positive whole number
  title:    z.string().trim().min(1, 'title is required').max(120), // 1–120 chars
  content:  z.string().trim().min(1, 'content is required'),        // not empty
});
```

**In plain English:**

- "When someone tries to create a note, `doctorId` must be a positive number,
  `title` must be 1–120 characters, `content` can't be empty."
- If the data breaks a rule, the request is rejected with **`400 Bad Request`**
  *before* it ever reaches the database.

**Key facts:**

- **When does it run?** On **every single request** to `POST /api/notes`.
  It's the "bouncer at the door."
- **Who wrote it?** You. These are your business rules.
- **Level:** the **application**. It guards what data is *allowed in*.

> Think of it as: **the security guard checking packages before they enter the
> warehouse.** Runs every time someone shows up.

---

## Side-by-side — this is the whole point

| | `migration.sql` | `noteSchema.js` |
|---|---|---|
| **Language** | SQL | JavaScript (Zod) |
| **Job** | Build the database tables | Check incoming request data |
| **Runs** | Once (at migration time) | Every request |
| **Level** | Database | Application |
| **Written by** | Auto-generated by Prisma | You |
| **Example rule** | `title TEXT NOT NULL` (column exists) | `title.min(1)` (not empty) + `.max(120)` |
| **On failure** | — (setup step) | Returns `400` to the user |

---

## "But they both say title is required — isn't that duplicated?"

This is exactly the confusion, and the answer is **no, they're two different safety nets:**

- **`migration.sql` → `NOT NULL`** = the *database* refuses to store a note with no
  title. The last line of defense.
- **`noteSchema.js` → `.min(1)`** = your *app* rejects it early, at the door, with a
  **friendly error message** (`"title is required"`) and the right status code — so
  the request never even reaches the database.

The database rule is a hard wall. The Zod rule is a polite bouncer who explains *why*
you can't come in. Good backends have **both**: validate early in the app (nice
errors), and let the database enforce it as a final guarantee.

---

## In the request flow

```
POST /api/notes  { title: "" }
        │
        ▼
  noteSchema.js  →  ❌ "title is required" → 400   ← stops HERE, DB never touched
        │ (if valid)
        ▼
  controller → prisma → INSERT into the Note table
                                    │
                                    ▼
                          migration.sql defined this table earlier
```

**Summary:**
`migration.sql` = the *shape* of the database (built once).
`noteSchema.js` = the *rules* for incoming data (checked every time).
